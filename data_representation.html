1. Three kinds of data

Every frame is a 433-dim vector made of three fundamentally different data types — they are not all rotations:

🔄

Rotation

All rotations are stored as continuous 6D rotation. Joint 0/pelvis becomes SMPL-X global_orient.

➡️

Translation

root trajectory · relative vs global
3
= Δx, Δz, height y
indices 330–332
x/z translationrelative to previous frame
y translationglobal root height per frame

x/z are integrated with cumulative sum to recover the global trajectory. y is used directly as absolute height.

😀

Expression

face · blendshape weights
100
= FLAME expr. coeffs
indices 333–432

Facial expression coefficients — weights over a learned face basis. Neither rotations nor positions.

Key point — there are NO 3D positions in the data. Joints are stored as rotations, and the body as a whole as translation. Actual 3D joint/vertex positions are computed only later through the SMPL-X model (decoded rotations + recovered root translation + body shape → mesh). So the model learns orientations and root motion, not joint coordinates.

4. How rotations are stored — why 6D?

Source SMPL-X .npz stores rotations as axis-angle (55×3). The loader (dataloaders/mix_sep.py:278) converts to continuous 6D rotation — friendlier for neural nets (no gimbal-lock / wrap-around):

axis-angle
55 × 3 = 165 → rotation matrix
axis_angle_to_matrix → 6D rotation
55 × 6 = 330 → + translation + face
= 433 → 3D positions
via forward kinematics
Global motion recovery: decoded trans_v[..., 0] and trans_v[..., 2] are integrated with cumulative sum to form global x/z root trajectory. Decoded trans_v[..., 1] is used directly as root height y. The decoded pelvis rotation becomes SMPL-X global_orient for each frame.

Normalization: mean_std/beatx_2_330_*.npy (rotation) & beatx_2_trans_*.npy (translation). Source: mix_sep.py __getitem__ lines 278–300, parts in rvq_beatx_train.py 175–258.

5. Decoder temporal field

The VQ decoder is non-causal, so one latent token can affect decoded frames both before and after its nominal frame.

Probe resultFramesMeaning
Past affected87Changing one middle latent token changes up to 87 earlier decoded frames.
Future affected90The same latent token changes up to 90 later decoded frames.
Total affected178The effective decoder field is larger than one 128-frame generation chunk.
Control implication: chunk-end guidance can report near-zero loss under chunk-local decoding, then move after full-sequence decoding because future latent tokens are inside the decoder field. Prepending previous latents helps the left boundary, but exact chunk-end control also needs right/future context.
